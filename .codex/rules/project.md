# Project Rules

## 基本方針

- このリポジトリは Slack へ株式市況を投稿する小さな Python Bot として扱う。
- 変更前に `README.md` と対象コードを確認し、既存の実行方法に合わせる。
- 不要なフレームワークや大きな構成変更を増やさず、標準ライブラリと既存依存を優先する。
- 小さな関数に分け、Slack投稿・データ取得・メッセージ整形をテストしやすい境界で保つ。

## 作業フロー

- セッション最初の実作業を開始するときは、現在の状態を確認してから新しい Git ブランチを作成する。
- 同じセッション内の追加依頼では、既存の作業ブランチを継続し、依頼ごとに新しいブランチを作らない。
- ブランチ名は原則として `codex/` prefix を使い、作業内容がわかる短い名前にする。
- ユーザーが特定のブランチ名や作業ブランチを指定した場合は、その指定を優先する。
- 既存の未コミット変更がある場合は、内容を確認し、ユーザーの変更を上書きしない。

## 実装方針

- 外部サービスへの通信は、タイムアウトとわかりやすいエラーメッセージを持たせる。
- Slackへ投稿する本文は、生成処理を分離して単体で確認できるようにする。
- 取得に失敗した銘柄や通信エラーがあっても、可能な範囲で原因がわかる表示にする。
- 依存ライブラリを増やす場合は、標準ライブラリや既存依存で足りない理由を確認してから `requirements.txt` に追加する。

## 実行とテスト

- 通常実行は `python main.py` または `uv run python main.py` を使う。
- 依存関係は `requirements.txt` を正とする。
- 挙動変更時は、可能な範囲で `python -m pytest` を実行する。
- テストでは実際の Slack Webhook や外部APIを呼ばず、モックや差し替え可能な関数を使う。
- `__pycache__`, `.pyc`, `.venv`, `.env` など生成物やローカル設定はコミットしない。

## ドキュメント

- 実行方法、環境変数、投稿内容、対象銘柄を変えたときは `README.md` も更新する。
- README の例では実在する秘密情報や実 Webhook URL を使わない。
- 現在の実装と README が食い違っている場合は、どちらを正とするか確認してから直す。

## 秘密情報

- Slack Incoming Webhook URL は `.env` または環境変数 `SLACK_WEBHOOK_URL` で扱う。
- `.env` や実 Webhook URL をコミットしない。
- README やテストに載せる URL は必ずダミー値にする。
